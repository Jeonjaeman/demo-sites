## 1단계 · 요청 분석

### 의뢰 핵심
80개 포트폴리오 기업을 해외·국내에 안내하는 영문(메인)/국문 웹사이트. **핵심 차별점은 각 기업이 발급받은 계정으로 직접 자사 정보를 업로드하는 다중 권한 CMS.** 예산 1,000만원 / 30일 / 1년 호스팅·유지보수 포함. 레퍼런스 = Y Combinator.

### 의뢰서에 없지만 반드시 필요한 것 (제안 포인트)

| # | 항목 | 근거 |
|---|------|------|
| 1 | **필터 축을 4개로 설계** (산업/투자단계/지역/상태) | 의뢰서엔 "검색·필터"만 명시. 벤치마킹한 5개사 전부 최소 2~4축을 씀. 데이터 스키마에 미리 넣지 않으면 재개발 발생 |

### 채택한 아이디어

**Index Ventures — 문장형 필터.** `Show me companies focused on [sector] based in [region]`. 필터 패널은 사용자가 질의를 조립하게 하지만, 문장은 질의를 손에 쥐여줍니다. 3개의 `<select>`로 구현 가능한데 기억에 남는 효과는 압도적입니다.

**Index / a16z / Kakao 공통 — 전량 DOM 렌더 후 클라이언트 필터링.** 80개(우리 규모)에서는 API 왕복이 순손실입니다. 즉시 반응하고, Ctrl+F가 되며, 크롤러가 전부 읽습니다. YC가 Algolia를 쓰는 건 5,000개이기 때문입니다. **~1,000개 미만에서는 이 방식이 명백히 우월합니다.**

**Sequoia — 투자 시점 vs 현재 상태 분리.** VC 사이트에서 가장 높은 레버리지.

**Sequoia / a16z / Altos 공통 — `border-radius: 0`.** 세 곳 모두 예외 없이 각진 모서리. SaaS가 아니라 기관으로 읽히게 만드는 단일 결정.

**전 세계 공통 — 볼드 없음.** YC(400), Sequoia(400), a16z(300), Altos(300). 위계는 크기와 서체로만.

### 의도적으로 하지 **않은** 것

- **a16z처럼 카드에 로고만 두기** — 로고를 모르면 카드가 막다른 길. a16z의 실제 약점.
- **Sequoia처럼 검색 없이 가기** — 415개에 검색창이 0개. 해외 사용자는 "robotics"로 찾습니다.
- **Index처럼 결과 수 없이 가기** — `AI/ML + Europe`이 조용히 0을 반환하고 복구 경로가 없음.

### ⚠️ 카카오벤처스 — 이번 프로젝트의 가장 중요한 반면교사

`kakao.vc/en/family`는 **UI만 번역하고 CMS 데이터는 한글로 남겼습니다.**
- 206개 카드 중 **이름 190개, 설명 205개가 여전히 한글**
- 한국어 페이지엔 **언어 전환기가 아예 없고**, 영문 페이지의 전환기는 `offsetWidth: 0` (보이지 않음)
- 한국어 218개 vs 영문 206개 — **12개가 조용히 사라짐**

해외 LP와 창업자가 보러 온 바로 그것(포트폴리오)이 영문 사이트에서 읽히지 않습니다. **의뢰서의 "영어 메인 + 한국어 지원"은 이 함정에 정확히 노출돼 있습니다.**

→ **대응:** 데이터 모델에서 기업별로 `tagline_en` / `tagline_ko`, `hq_en` / `hq_ko`를 **분리 필드로 강제**하고, 국문 누락 시 영문으로 폴백합니다. 언어 전환기는 **양쪽 로케일 모두**에서, 헤더와 푸터 두 곳에 보이게 배치했습니다.

**Altos의 반대 교훈:** 알토스는 포트폴리오 144개 중 83개가 아시아인 한국 중심 펀드인데도 **영문 전용**을 선택했습니다. 하나의 정본 사이트, 방치될 한국어 트리 없음. **"영문 전용"과 "제대로 된 이중언어" 사이에서 의도적으로 고르십시오. 최악은 전환기가 있는데 그 너머가 부실한 것입니다.**

## 4단계 · 구현

```
index.html              섹션 10개
assets/css/style.css    디자인 시스템 + 컴포넌트
assets/js/data.js       80개 기업 (CMS API 응답과 동일 형태)
assets/js/i18n.js       영/국문 사전 95:95
assets/js/app.js        필터 엔진 · 다국어 · 드로어 · 리빌
```

**섹션 구성:** 헤더 → 히어로 → 통계 → 소개 → 투자철학 4가지 → **포트폴리오** → **기업 포털** → 뉴스 → CTA → 푸터

**포트폴리오 = 벤치마킹 종합:**
문장형 필터(Index) + 자유 검색(YC) + 옵션별 카운트가 붙은 레일(YC) + 상태 배지(Sequoia) + 전량 DOM 필터링(a16z/Kakao) + Load More(Sequoia) + **원클릭 복구가 가능한 빈 상태**(Index의 결함 보완).

**한국어 처리:** Outfit에 한글 글리프가 없어 Pretendard로 자연 폴백. 한글은 상승/하강부가 없고 광학적 크기가 커서 `html[lang="ko"]`에서 행간을 넓히고(1.75) 음수 자간을 완화하며 H1 스케일을 44px→36px로 낮춥니다.

**문장형 필터의 어순 문제:** 영어는 동사가 앞, 한국어는 뒤. DOM 순서를 고정한 채 로케일별 연결어 슬롯 `[a] 산업 [b] 지역 [c] 단계 [d]`로 해결했습니다.
- EN: `Show me companies focused on [Fintech] based in [Korea] at [Series A]`
- KO: `[핀테크], [한국], [시리즈 A] 기업을 보여주세요.`

### 검증에서 실제로 잡아낸 결함

1. **CTA 섹션 정렬 붕괴** — `.cta__in`이 `.wrap`(gutter)과 `max-width`를 동시에 가져 `margin-inline:auto`에 가운데 정렬됨. 이 섹션만 텍스트가 25px 안으로 밀려 있었습니다. → 제약을 자식으로 이동.
2. **YC 팔레트의 대비 미달** — `--muted: #8A8575`가 크림 배경 대비 **3.37:1**로 AA 미달. 카드 설명·결과 수·푸터 등 다수 실패. → `#6E6A5C`로 보정(YC 원본도 동일 결함).
3. **강조색 오렌지가 소형 텍스트에 부적합** — `#FF5100`이 11px에서 2.8:1. → 채움용 `--accent`와 텍스트용 `--accent-txt: #B03A00`(5.56:1)로 분리. `pill--accent`는 흰 글자(3.27) → 잉크 글자(5.63).
4. **로고 틴트 3개 미달** — 흰 글자 기준 오렌지 3.27 / 초록 4.37 / 앰버 3.58 → 각각 보정.
5. **JS 실패 시 백지** — `.reveal`이 `opacity:0`으로 시작해 스크립트가 죽으면 사이트 전체가 영구히 안 보임. → `.no-js`/`.js` 스코프 + 관찰자 미발화 시 1,200ms 안전장치. 통계 수치도 마크업에 실값을 넣어 무JS에서도 표시.
6. **국문 "팀" 링크 13px 폭** — 최소 타겟 미달. → 레이블을 "팀 소개"로(읽기에도 자연스러움).

# 확장 작업 (2차)

## ① 기업 상세 개별 페이지 — SEO 표면

`build.js`가 `data.js` 하나에서 **164페이지를 정적 생성**합니다.

```
companies/<slug>.html      companies/ko/<slug>.html    80 × 2
team.html                  ko/team.html
about.html                 ko/about.html
sitemap.xml (165 URL)      robots.txt (/cms/ 차단)
```

**클라이언트 토글이 아니라 로케일별 실제 URL을 쓴 이유:** 크롤러는 문서당 하나의 URL을 인덱싱합니다. 카카오벤처스처럼 토글 하나로 처리하면 한국어 콘텐츠가 검색에 잡히지 않습니다. 각 페이지는 canonical + 상호 hreflang(en/ko/x-default) + OG locale + JSON-LD `Organization`을 갖습니다.

**단, 이 방식은 양쪽 트리에 진짜 번역이 있을 때만 의미가 있습니다.** 실측:

| | EN 트리 | KO 트리 |
|---|---|---|
| 페이지 | 80 | 80 (**세트 동일 — 카카오는 218 vs 206**) |
| 한글 비중 | 0% (전환기 "한국어" 3자만) | **72%** (나머지는 기업명 등 고유명사) |
| 국문 카피 없는 페이지 | — | **0 / 80** (카카오: 205/206이 실패) |

드로어는 유지하되(스크롤·필터 상태 보존) **퍼머링크를 함께 제공**합니다 — a16z 방식. 로케일에 따라 `companies/` ↔ `companies/ko/`로 분기합니다.

**검증 중 잡은 결함**
- **한국어 조사 자동화 불가** — `"Nexlify은(는) 이후 기업공개를 완료했습니다."` 영문 고유명사 뒤 은/는은 받침 유무에 달려 판정이 불가능합니다. 주어를 `이 회사는`으로 치환해 조사를 제거했습니다. 조사 병기 잔여 **0건**.
- **`ol` 리셋 누락** — 리셋이 `ul`만 처리해 `<ol>`이 브라우저 기본 `padding-left: 40px`를 물고 있었습니다. 브레드크럼과 **`index.html`의 기업 포털 3단 그리드(`<ol class="steps">`)가 40px 밀려** 있었습니다. 앞선 감사가 놓친 이유는 컨테이너 좌측만 쟀기 때문 — 이후 **Range API로 실제 텍스트 좌측**을 재도록 감사를 고쳤습니다.

## ② 관리자 / 기업 CMS 목업

```
cms/index.html    로그인 (자격증명 미수집·미전송)
cms/admin.html    최고 관리자 — 계정 발급 · 검수 큐
cms/company.html  투자사 관리자 — 프로필 편집 + 실시간 미리보기
```

의뢰서의 두 축을 화면으로 증명합니다.

**최고 관리자** — 통계(계정 80 = 게시 66 + 대기 7 + 미작성 7), 검수 큐(승인/반려 시 카운트 즉시 이동), 계정 일괄 생성(이메일 형식 검증 실패 시 버튼 차단), 기업명·이메일 검색. 목록은 `div` 그리드가 아닌 **시맨틱 `<table>`**(`scope`, `aria-sort`) — Sequoia 방식.

**투자사 관리자** — 1단계 분석에서 지적한 **입력 규격화**가 실제로 작동합니다:
- 한 줄 소개 60자 초과 시 카운터·입력창·미리보기가 **동시에** 경고
- 로고는 1:1 고정 + 자동 크롭 안내
- **영문/국문이 각각 독립 필드**이고 탭마다 입력 여부 점(초록/주황)이 표시됩니다 — 카카오의 실패는 CMS에 기업당 텍스트 필드가 **하나뿐**이어서 생겼습니다
- 국문을 비우면 미리보기가 영문으로 폴백되며 **왜 그런지 문장으로 설명**
- 영문 미입력 시 제출 차단(영문이 기본 언어)
- 저장 → 배너가 "게시 중" → "검수 대기"로 전환

미리보기는 **공개 그리드의 `.card` 마크업을 그대로** 렌더링합니다. 80명이 각자 입력해도 그리드가 깨지지 않는 근거입니다.

**검증 중 잡은 결함**
- **성공 메시지 초록 3.9:1** — `--ok`/`--warn`은 채움으로는 통과(흰 글자 4.59/4.53)하지만 텍스트로는 미달. 강조색과 동일하게 `--ok-txt`(#267A37) / `--warn-txt`(#A05A00) 쌍으로 분리했습니다. `--stop`은 6.58로 이미 통과라 그대로.
- **국문 입력 3개에 label 없음** — 보이는 label이 영문 필드에 묶여 있어, 스크린리더에서 **폼의 한국어 절반이 이름 없는 입력창**으로 읽혔습니다. `.sr` label 추가.

## ③ 팀 · 소개 페이지

**팀** — 파트너 8인. Sequoia의 **파트너를 포트폴리오 축으로 쓰는 방식**을 가져와, 각 파트너가 담당 기업으로 직접 링크됩니다(80개 전량 귀속, 미배정 0, 모든 기업이 담당 파트너의 focus와 일치). 인재 파트너는 담당 기업 0 + "투자 결정에 참여하지 않습니다"로 표시 — 로스터를 장식이 아니라 창업자가 쓸 수 있는 귀속 그래프로 만듭니다. 기업 상세의 "담당 파트너" 항목은 `team.html#<slug>` 앵커로 연결됩니다.

알토스는 팀 페이지가 **아예 없는데**, 해외 LP가 기대하는 항목입니다.

**소개** — 연혁 6개(2011 → 2026), 투자 원칙 4개, 사무소 4곳. 연혁은 연도/내용 2단이며 실측상 연도 전부 24px·제목 전부 176px로 정렬됩니다.

홈은 클라이언트 토글이라 `data-href-en/ko`로 **활성 로케일에 따라 링크를 전환**합니다(EN → `about.html`, KO → `ko/about.html`).

## ④ 홈 로케일 분리 — 마지막 SEO 공백 제거

앞선 2차까지는 **홈만 클라이언트 토글로 남아 있어 국문 홈이 인덱싱되지 않았습니다.** 홈은 검색에서 가장 중요한 문서인데 한국어 URL이 아예 존재하지 않았습니다.

**구현 방식** — 300줄 마크업을 JS 문자열로 옮기면 클라이언트 개발팀이 유지보수할 수 없으므로, **`index.html`을 영문 페이지 겸 소스로 두고 `build.js`가 이를 변환**해 `ko/index.html`을 굽습니다.

| 변환 | 처리 |
|---|---|
| `data-i18n` / `-ph` / `-aria` | 사전 값으로 치환 (사전에 없는 키는 **빌드 실패**로 처리) |
| 깊이 | `assets/`·`cms/` → `../` 접두. `about.html`·`team.html`은 **그대로** — `ko/about.html`이 형제라 같은 문자열이 양쪽에서 동작 |
| 언어 전환기 | `<button>` → **실제 링크**, `is-on` 이동 |
| head | lang·title·description·canonical·og:locale 교체, `data-up="../"` 주입 |

`app.js`는 이제 **로케일을 `document.documentElement.lang`에서 읽습니다**(localStorage·토글 제거). 정적 카피는 빌드가 굽고, 동적 렌더(카드·필터·문장·드로어)만 사전을 씁니다. 기업 링크는 `data-up`으로 깊이를 보정합니다.

**검증 중 잡은 결함 — CSS 특정성 충돌**
`<button class="lang__btn">`을 `<a>`로 바꾸자 푸터 전환기가 **`.ft__col a`(0,1,1)에 걸려 `.lang__btn`(0,1,0)의 패딩을 빼앗겼습니다.** 헤더는 33px인데 푸터만 23px로 최소 타겟 미달. `.ft__col a:not(.lang__btn)`으로 분리했습니다.

> **측정 함정:** 이 결함을 처음엔 재현하지 못했습니다. 브라우저가 `style.css`를 캐시해 **수정 전 CSS로 측정**하고 있었기 때문입니다(`force` 네비게이션도 하위 리소스 캐시는 무효화하지 않음). 서버 응답과 브라우저 스타일시트를 대조해 확인한 뒤, 이후 감사는 스타일시트를 캐시 무력화한 상태에서 재실행했습니다.

## 실서비스 이관 시 참고

- `data.js`의 `PORTFOLIO` 배열은 **CMS API(`GET /api/portfolio`) 응답과 동일한 형태**입니다. `fetch`로 교체하면 프론트 수정 없이 붙습니다. `PARTNERS`도 동일.
- **`build.js`가 그대로 배포 파이프라인이 됩니다.** 기업이 프로필을 승인받을 때마다 재실행하면 해당 기업의 2개 페이지와 사이트맵이 갱신됩니다. CI에서 `node build.js` 한 줄.
- 80개 규모에서는 전량 렌더 + 클라이언트 필터가 최적입니다. **500~1,000개를 넘어가면** 서버 페이징/Algolia를 재검토하십시오.
- **클라이언트 토글은 전부 제거됐습니다.** 모든 페이지가 로케일별 실제 URL이며, 언어 전환기는 링크입니다. `index.html`은 영문 페이지이자 `ko/index.html`의 소스이므로 **홈 수정은 `index.html`에서 하고 `node build.js`를 다시 실행**하십시오.
- `index.html`의 `data-i18n` 속성은 지우지 마십시오 — 빌드가 이 속성을 기준으로 국문을 치환합니다. 사전에 없는 키를 쓰면 빌드가 오류로 알려줍니다.
- CMS는 목업입니다. 로그인은 자격증명을 수집·전송하지 않으며, 검수 승인·계정 생성은 화면 안에서만 반영됩니다. `robots.txt`에서 `/cms/`를 차단해 두었습니다.
- 로고는 이미지 없이 이니셜+틴트로 대체했습니다. 실제 로고 도입 시 **a16z의 `invert`/`brightness` CSS 필터 기법**을 권장합니다 — 기업당 SVG 하나로 밝은/어두운 배경 양쪽을 처리하며, "흰색 버전 로고 보내주세요" 요청이 사라집니다. 기업 포털의 로고 안내 문구는 이미 이 전제로 쓰여 있습니다.
- 색 토큰은 **채움/텍스트 쌍**으로 나뉘어 있습니다(`--accent`/`--accent-txt`, `--ok`/`--ok-txt`, `--warn`/`--warn-txt`). 한 색이 흰 글자를 받으면서 동시에 크림 위 소형 텍스트로도 읽히는 건 물리적으로 불가능합니다 — 새 색을 추가할 때 이 규칙을 지키십시오.
